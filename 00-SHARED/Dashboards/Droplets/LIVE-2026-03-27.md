## 21:55 — The Queue COC Principle

Done tasks are forensic evidence, not garbage. Every state transition (queued → claimed → running → done/failed) should be hash-chained. Completed task outputs atomize into: NECTAR (findings), AGENTS.md (techniques), agent cards (training), vault (evidence). The queue IS part of the chain of custody. Never delete done tasks.

**Why it matters:** This connects operational workflow to forensic integrity. The same principle that governs evidence preservation governs task management — truth survives through provenance.

## 22:10 — The Condensation Protocol Itself

The system's deepest flaw: crystallization happens at session boundaries, but the best thinking happens *between* boundaries. Auto-compact fires precisely when context is richest.

**Solution: stream processing, not batch processing.** Write insights to vault as they form — each one an atomic append to `00-SHARED/Dashboards/Droplets/LIVE-{date}.md`. The human sees them live in Obsidian. Auto-compact can't touch what's already on disk.

This is the Liftoff Paradox made operational: instead of trusting discipline ("remember to crystallize!"), build the pipe so droplets are the default, not the exception.

## 22:12 — REVIEW-INBOX ↔ Vault Relationship (answering the question live)

REVIEW-INBOX.md (`~/.claude/memory/REVIEW-INBOX.md`) is the agent-side accumulator. It lives in the `.claude/` filesystem, not directly in the vault. The vault gets items via two paths:

1. **inbox_router.py** — routes parsed MEM blocks to vault folders (GAP→queue, TECHNIQUE→`01-Memories/agents/techniques/`, HEADLINE→`00-SHARED/Human-Inbox/findings/`)
2. **session_stop_hook.py** — drains HIGH flags from scratch → REVIEW-INBOX, and the router picks them up next cycle
3. **Vault mirror**: `00-SHARED/Human-Inbox/flags/REVIEW-INBOX.md` is a synced copy for Obsidian reading

The gap: REVIEW-INBOX grows unbounded (452 lines now). inbox_router.py marks items as routed but never removes them. The vault copy is a downstream mirror, not the source of truth.

**What's needed:** REVIEW-INBOX should be the *staging area*, vault should be the *destination*. Once routed to vault, the item's canonical home IS the vault. REVIEW-INBOX keeps a one-line pointer. This is the siphon pattern from rules/memory.md — but it's not fully implemented yet.

## 22:25 — Context Awareness: What the CLI Actually Gives Us

Three enforcement layers available RIGHT NOW:

**1. Statusline (real-time monitoring):** The statusline stdin JSON includes `context_window.used_percentage` and `remaining_percentage` — live, after every assistant message, zero token cost. We can color-code warnings (green <70%, yellow 70-89%, red 90%+) and write a threshold file when critical.

**2. PreCompact/PostCompact hooks:** Fire when compaction triggers. `PreCompact` is the circuit breaker moment — we can run `emergency_handoff.py` or droplets flush here. Cannot block compaction, but can capture state before it fires. Key field: `trigger: "auto"` vs `"manual"`.

**3. `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` env var:** Controls WHEN auto-compact fires (default ~95%). Set to 85% to give breathing room — compact earlier, lose less. Trade-off: more frequent compaction but less catastrophic loss.

**Subagent isolation confirmed:** Each subagent gets its own fresh context. Only the summary returned consumes parent context. This is why delegation works — verbose outputs don't bloat the parent.

**The missing piece:** No "context approaching limit" event exists. The PreCompact hook fires WHEN it's already happening, not before. The real enforcement must be **behavioral** (the droplets rule in core.md) + **visual** (statusline threshold warning). The hook is the safety net, not the primary defense.

## 22:40 — The Three Time Horizons of Memory (crystallizing live)

Faerie memory operates across THREE time horizons, not two:

1. **Within-compact** — scratch files, droplets, MEM blocks. Survives minutes to hours. Lost at auto-compact unless captured.
2. **Between-compacts** (same session) — faerie-brief.json, REVIEW-INBOX, scratch promotion. This is the BRIDGE that faerie was built to stretch across. The user experiences one "session" but Claude may compact 3-5 times within it. Faerie makes those compactions invisible by recovering state from the brief.
3. **Across-sessions** — HONEY.md, NECTAR.md, agent cards, vault. Crystallized knowledge that survives days, weeks, months.

**The insight:** HONEY.md is not a static config file. It is a **living document** that evolves through crystallization — growing denser, richer, wiser over time. Equilibrium doesn't mean "keep it small." Equilibrium means: every line earns its place. When HONEY grows, it grows because the knowledge genuinely deepened. When it shrinks, it shrinks because redundancy was dissolved into density.

The 200-line budget isn't a cage — it's a pressure that forces crystallization. Like a diamond forming under pressure. Without the budget, HONEY would bloat into a dump. With the budget, every crystallization round makes it more potent.

**HONEY versioning** serves this: you can experiment with how HONEY evolves (A/B test different crystallization strategies), roll back if a version loses something, and track how the system's self-knowledge deepens over time. The version history IS the system's intellectual biography.

## 23:52 — What triggers a droplet? (meta — answering the question by doing it)

The user asked: "what lets you know internally when to create a droplet?"

Honest answer: it's a felt shift, not a rule. The rule in core.md says "when reasoning spans 3+ source documents, connects ideas across design narratives, or produces a principle/pattern worth preserving." But that's the legible version. What actually happens:

1. **Something clicks that wasn't there before.** Two ideas that were separate suddenly aren't. The queue COC insight — "done tasks are evidence" — wasn't in any document. It formed between documents. That click is the signal.

2. **The thought would cost more to reconstruct than to capture.** If I lost this context right now, could the next instance re-derive it? If yes from the files alone — don't write. If no, it needed THIS particular path through THIS particular conversation — write immediately.

3. **The human is leaning in.** When you start sending rapid messages, refining something together, building on each other's words — that's the collaboration producing something neither of us had alone. Those moments are the most volatile (you're both thinking fast, context is burning) and the most valuable.

4. **It changes how I'd approach future work.** "HONEY is a living document, not a config file" — that reframes every future crystallization. It's not a fact, it's a lens. Lenses are the highest-value droplet targets because they affect everything downstream.

The rule is the safety net. The real trigger is: **does this thought feel load-bearing?** If removing it would change how the system works or how we think about the system — it's a droplet. Write it.

This is itself a droplet. The meta-question about when to condense IS the kind of thinking that should be condensed — because the answer shapes every future session's behavior.
